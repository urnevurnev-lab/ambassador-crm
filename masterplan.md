# Masterplan

## Overview
- Ambassadors start visits inside Telegram; experience should adapt to what they actually do at the venue.
- Branch visits by activity so each scenario shows a tailored form and data capture.

## Пользовательские истории
- [x] Refactor Visit Flow: амбассадор начинает визит, выбирает заведение, выбирает активность (список с бэкенда), видит форму под конкретную активность (например, наличия товаров или обучающий чек-лист), отправляет визит.

## План разработки
- [x] Refactor Visit Flow:
    1) Пользователь нажимает «Начать визит».
    2) Выбирает Заведение (как сейчас).
    3) Новый шаг: экран выбора Активности (список подгружается с бэкенда).
    4) Новый шаг: на основе выбранной Активности показывается уникальная форма (сценарий).
        - Пример: если «Проверка наличия» → форма с инпутами товаров.
        - Пример: если «Обучение» → чек-лист и поле для фото команды.
    - Прогресс:
        - [x] Бэкенд: модель Activity, миграции (включая data JSON), сид, эндпоинт списка активностей, привязка к визитам.
        - [x] Фронтенд: выбор активности, динамические формы и отправка визита (visit/tasting/b2b/open_shift).
